1//! Minimalistic testing framework for generating tests for a given set of test cases
2//! and decorating them to add retries, timeouts, sequential test processing etc. In other words,
3//! the framework implements:
4//!
5//! - Parameterized tests of reasonably low cardinality for the standard Rust test runner
6//! - Fully code-based, composable and extensible test decorators.
7//!
8//! # Overview
9//!
10//! ## Test cases
11//!
12//! [`test_casing`](macro@test_casing) attribute macro wraps a free-standing function
13//! with one or more arguments and transforms it into a collection of test cases.
14//! The arguments to the function are supplied by an iterator (more precisely,
15//! an expression implementing [`IntoIterator`]).
16//!
17//! For convenience, there is [`TestCases`], a lazy iterator wrapper that allows constructing
18//! test cases which cannot be constructed in compile time (e.g., ones requiring access to heap).
19//! [`TestCases`] can be instantiated using the [`cases!`] macro.
20//!
21//! Since a separate test wrapper is generated for each case, their number should be
22//! reasonably low (roughly speaking, no more than 20).
23//! Isolating each test case makes most sense if the cases involve some heavy lifting
24//! (spinning up a runtime, logging considerable amount of information, etc.).
25//!
26//! ## Test decorators
27//!
28//! [`decorate`] attribute macro can be placed on a test function to add generic functionality,
29//! such as retries, timeouts or running tests in a sequence.
30//!
31//! The [`decorators`] module defines some basic decorators and the
32//! [`DecorateTest`](decorators::DecorateTest) trait allowing to define custom decorators.
33//! Test decorators support async tests, tests returning `Result`s and test cases; see
34//! the module docs for more details.
35//!
36//! # Test cases structure
37//!
38//! The generated test cases are placed in a module with the same name as the target function
39//! near the function.
40//! This allows specifying the (potentially qualified) function name to restrict the test scope.
41//!
42//! If the [`nightly` crate feature](#nightly) is not enabled, names of particular test cases
43//! are not descriptive; they have the `case_NN` format, where `NN` is the 0-based case index.
44//! The values of arguments provided to the test are printed to the standard output
45//! at the test start. (The standard output is captured and thus may be not visible
46//! unless the `--nocapture` option is specified in the `cargo test` command.)
47//!
48//! If the `nightly` feature *is* enabled, the names are more descriptive, containing [`Debug`]
49//! presentation of all args together with their names. Here's an excerpt from the integration
50//! tests for this crate:
51//!
52//! ```text
53//! test number_can_be_converted_to_string::case_1 [number = 3, expected = "3"] ... ok
54//! test number_can_be_converted_to_string::case_2 [number = 5, expected = "5"] ... ok
55//! test numbers_are_large::case_0 [number = 2] ... ignored, testing that `#[ignore]` attr works
56//! test numbers_are_large::case_1 [number = 3] ... ignored, testing that `#[ignore]` attr works
57//! test string_conversion_fail::case_0 [bogus_str = "not a number"] - should panic ... ok
58//! test string_conversion_fail::case_1 [bogus_str = "-"] - should panic ... ok
59//! test string_conversion_fail::case_2 [bogus_str = ""] - should panic ... ok
60//! ```
61//!
62//! The names are fully considered when filtering tests, meaning that it's possible to run
63//! particular cases using a filter like `cargo test 'number = 5'`.
64//!
65//! # Alternatives and similar tools
66//!
67//! - The approach to test casing from this crate can be reproduced with some amount of copy-pasting
68//!   by manually feeding necessary inputs to a common parametric testing function.
69//!   Optionally, these tests may be collected in a module for better structuring.
70//!   The main downside of this approach is the amount of copy-pasting.
71//! - Alternatively, multiple test cases may be run in a single `#[test]` (e.g., in a loop).
72//!   This is fine for the large amount of small cases (e.g., mini-fuzzing), but may have downsides
73//!   such as overflowing or overlapping logs and increased test runtimes.
74//! - The [`test-case`] crate uses a similar approach to test case structuring, but differs
75//!   in how test case inputs are specified. Subjectively, the approach used by this crate
76//!   is more extensible and easier to read.
77//! - [Property testing] / [`quickcheck`]-like frameworks provide much more exhaustive approach
78//!   to parameterized testing, but they require significantly more setup effort.
79//! - [`rstest`] supports test casing and some of the test decorators (e.g., timeouts).
80//! - [`nextest`] is an alternative test runner that supports most of the test decorators
81//!   defined in the [`decorators`] module. It does not use code-based decorator config and
82//!   does not allow for custom decorator.
83//!
84//! [`test-case`]: https://docs.rs/test-case/
85//! [Property testing]: https://docs.rs/proptest/
86//! [`quickcheck`]: https://docs.rs/quickcheck/
87//! [`rstest`]: https://crates.io/crates/rstest
88//! [`nextest`]: https://nexte.st/
89//!
90//! # Crate features
91//!
92//! ## `nightly`
93//!
94//! *(Off by default)*
95//!
96//! Uses [custom test frameworks] APIs together with a generous spicing of hacks
97//! to include arguments in the names of the generated tests (see an excerpt above
98//! for an illustration). `test_casing` actually does not require a custom test runner,
99//! but rather hacks into the standard one; thus, the generated test cases can run alongside with
100//! ordinary / non-parameterized tests.
101//!
102//! Requires a nightly Rust toolchain and specifying `#![feature(test, custom_test_frameworks)]`
103//! in the using crate. Because `custom_test_frameworks` APIs may change between toolchain releases,
104//! the feature may break. See [the CI config] for the nightly toolchain version the crate
105//! is tested against.
106//!
107//! ## `tracing`
108//!
109//! *(Off by default)*
110//!
111//! Enables tracing / logging using [the `tracing` library](https://docs.rs/tracing/). This includes
112//! the [`Trace`](decorators::Trace) decorator.
113//!
114//! [custom test frameworks]: https://github.com/rust-lang/rust/issues/50297
115//! [the CI config]: https://github.com/slowli/test-casing/blob/main/.github/workflows/ci.yml

126/// Wraps a tested function to add retries, timeouts etc.
127///
128/// # Inputs
129///
130/// This attribute must be placed on a test function (i.e., one decorated with `#[test]`,
131/// `#[tokio::test]`, etc.). The attribute must be invoked with a comma-separated list
132/// of one or more [test decorators](decorators::DecorateTest). Each decorator must
133/// be a constant expression (i.e., it should be usable as a definition of a `static` variable).
134///
135/// # Examples
136///
137/// ## Basic usage
138///
139/// ```no_run
140/// use test_casing::{decorate, decorators::Timeout};
141///
142/// #[test]
143/// #[decorate(Timeout::secs(1))]
144/// fn test_with_timeout() {
145///     // test logic
146/// }
147/// ```
148///
149/// ## Tests returning `Result`s
150///
151/// Decorators can be used on tests returning `Result`s, too:
152///
153/// ```no_run
154/// use test_casing::{decorate, decorators::{Retry, Timeout}};
155/// use std::error::Error;
156///
157/// #[test]
158/// #[decorate(Timeout::millis(200), Retry::times(2))]
159/// // ^ Decorators are applied in the order of their mention. In this case,
160/// // if the test times out, errors or panics, it will be retried up to 2 times.
161/// fn test_with_retries() -> Result<(), Box<dyn Error + Send>> {
162///     // test logic
163/// #   Ok(())
164/// }
165/// ```
166///
167/// ## Multiple `decorate` attributes
168///
169/// Multiple `decorate` attributes are allowed. Thus, the test above is equivalent to
170///
171/// ```no_run
172/// # use test_casing::{decorate, decorators::{Retry, Timeout}};
173/// # use std::error::Error;
174/// #[test]
175/// #[decorate(Timeout::millis(200))]
176/// #[decorate(Retry::times(2))]
177/// fn test_with_retries() -> Result<(), Box<dyn Error + Send>> {
178///     // test logic
179/// #   Ok(())
180/// }
181/// ```
182///
183/// ## Async tests
184///
185/// Decorators work on async tests as well, as long as the `decorate` macro is applied after
186/// the test macro:
187///
188/// ```no_run
189/// # use test_casing::{decorate, decorators::Retry};
190/// #[tokio::test]
191/// #[decorate(Retry::times(3))]
192/// async fn async_test() {
193///     // test logic
194/// }
195/// ```
196///
197/// ## Composability and reuse
198///
199/// Decorators can be extracted to a `const`ant or a `static` for readability, composability
200/// and/or reuse:
201///
202/// ```no_run
203/// # use test_casing::{decorate, decorators::*};
204/// # use std::time::Duration;
205/// const RETRY: RetryErrors<String> = Retry::times(2)
206///     .with_delay(Duration::from_secs(1))
207///     .on_error(|s| s.contains("oops"));
208///
209/// static SEQUENCE: Sequence = Sequence::new().abort_on_failure();
210///
211/// #[test]
212/// #[decorate(RETRY, &SEQUENCE)]
213/// fn test_with_error_retries() -> Result<(), String> {
214///     // test logic
215/// #   Ok(())
216/// }
217///
218/// #[test]
219/// #[decorate(&SEQUENCE)]
220/// fn other_test() {
221///     // test logic
222/// }
223/// ```
224///
225/// ## Use with `test_casing`
226///
227/// When used together with the [`test_casing`](macro@test_casing) macro, the decorators will apply
228/// to each generated case.
229///
230/// ```
231/// use test_casing::{decorate, test_casing, decorators::Timeout};
232///
233/// #[test_casing(3, [3, 5, 42])]
234/// #[decorate(Timeout::secs(1))]
235/// fn parameterized_test_with_timeout(input: u64) {
236///     // test logic
237/// }
238/// ```

240/// Flattens a parameterized test into a collection of test cases.
241///
242/// # Inputs
243///
244/// This attribute must be placed on a freestanding function with 1..8 arguments.
245/// The attribute must be invoked with 2 values:
246///
247/// 1. Number of test cases, a number literal
248/// 2. A *case iterator* expression evaluating to an implementation of [`IntoIterator`]
249///    with [`Debug`]gable, `'static` items.
250///    If the target function has a single argument, the iterator item type must equal to
251///    the argument type. Otherwise, the iterator must return a tuple in which each item
252///    corresponds to the argument with the same index.
253///
254/// A case iterator expression may reference the environment (e.g., it can be a name of a constant).
255/// It doesn't need to be a constant expression (e.g., it may allocate in heap). It should
256/// return at least the number of items specified as the first attribute argument, and can
257/// return more items; these additional items will not be tested.
258///
259/// [`Debug`]: core::fmt::Debug
260///
261/// # Mapping arguments
262///
263/// To support more idiomatic signatures for parameterized test functions, it is possible
264/// to *map* from the type returned by the case iterator. The only supported kind of mapping
265/// so far is taking a shared reference (i.e., `T` → `&T`). The mapping is enabled by placing
266/// the `#[map(ref)]` attribute on the corresponding argument. Optionally, the reference `&T`
267/// can be further mapped with a function / method (e.g., `&String` → `&str` with
268/// [`String::as_str()`]). This is specified as `#[map(ref = path::to::method)]`, a la
269/// `serde` transforms.
270///
271/// # Examples
272///
273/// ## Basic usage
274///
275/// `test_casing` macro accepts 2 args: number of cases and the iterator expression.
276/// The latter can be any valid Rust expression.
277///
278/// ```
279/// # use test_casing::test_casing;
280/// #[test_casing(5, 0..5)]
281/// // #[test] attribute is optional and is added automatically
282/// // provided that the test function is not `async`.
283/// fn number_is_small(number: i32) {
284///     assert!(number < 10);
285/// }
286/// ```
287///
288/// Functions returning `Result`s are supported as well.
289///
290/// ```
291/// # use test_casing::test_casing;
292/// use std::error::Error;
293///
294/// #[test_casing(3, ["0", "42", "-3"])]
295/// fn parsing_numbers(s: &str) -> Result<(), Box<dyn Error>> {
296///     let number: i32 = s.parse()?;
297///     assert!(number.abs() < 100);
298///     Ok(())
299/// }
300/// ```
301///
302/// The function on which the `test_casing` attribute is placed can be accessed from other code
303/// (e.g., for more tests):
304///
305/// ```no_run
306/// # use test_casing::test_casing;
307/// # use std::error::Error;
308/// #[test_casing(3, ["0", "42", "-3"])]
309/// fn parsing_numbers(s: &str) -> Result<(), Box<dyn Error>> {
310///     // snipped...
311/// #   Ok(())
312/// }
313///
314/// #[test]
315/// fn parsing_number_error() {
316///     assert!(parsing_numbers("?").is_err());
317/// }
318/// ```
319///
320/// The number of test cases is asserted in a separate generated test. This ensures that no cases
321/// are skipped by "underreporting" this count (e.g., if the test cases expression is extended).
322///
323/// ## Case expressions
324///
325/// Case expressions can be extracted to a constant for reuse or better code structuring.
326///
327/// ```
328/// # use test_casing::{cases, test_casing, TestCases};
329/// const CASES: TestCases<(String, i32)> = cases! {
330///     [0, 42, -3].map(|i| (i.to_string(), i))
331/// };
332///
333/// #[test_casing(3, CASES)]
334/// fn parsing_numbers(s: String, expected: i32) {
335///     let parsed: i32 = s.parse().unwrap();
336///     assert_eq!(parsed, expected);
337/// }
338/// ```
339///
340/// This example also shows that semantics of args is up to the writer; some of the args may be
341/// expected values, etc.
342///
343/// ## Cartesian product
344///
345/// One of possible case expressions is a [`Product`]; it can be used to generate test cases
346/// as a Cartesian product of the expressions for separate args.
347///
348/// ```
349/// # use test_casing::{test_casing, Product};
350/// #[test_casing(6, Product((0_usize..3, ["foo", "bar"])))]
351/// fn numbers_and_strings(number: usize, s: &str) {
352///     assert!(s.len() <= number);
353/// }
354/// ```
355///
356/// ## Reference args
357///
358/// It is possible to go from a generated argument to its reference by adding
359/// a `#[map(ref)]` attribute on the argument. The attribute may optionally specify
360/// a path to the transform function from the reference to the desired type
361/// (similar to transform specifications in the [`serde`](https://docs.rs/serde/) attr).
362///
363/// ```
364/// # use test_casing::{cases, test_casing, TestCases};
365/// const CASES: TestCases<(String, i32)> = cases! {
366///     [0, 42, -3].map(|i| (i.to_string(), i))
367/// };
368///
369/// #[test_casing(3, CASES)]
370/// fn parsing_numbers(#[map(ref)] s: &str, expected: i32) {
371///     // Snipped...
372/// }
373///
374/// #[test_casing(3, CASES)]
375/// fn parsing_numbers_too(
376///     #[map(ref = String::as_str)] s: &str,
377///     expected: i32,
378/// ) {
379///     // Snipped...
380/// }
381/// ```
382///
383/// ## `ignore` and `should_panic` attributes
384///
385/// `ignore` or `should_panic` attributes can be specified below the `test_casing` attribute.
386/// They will apply to all generated tests.
387///
388/// ```
389/// # use test_casing::test_casing;
390/// #[test_casing(3, ["not", "implemented", "yet"])]
391/// #[ignore = "Promise this will work sometime"]
392/// fn future_test(s: &str) {
393///     unimplemented!()
394/// }
395///
396/// #[test_casing(3, ["not a number", "-", ""])]
397/// #[should_panic(expected = "ParseIntError")]
398/// fn string_conversion_fail(bogus_str: &str) {
399///     bogus_str.parse::<i32>().unwrap();
400/// }
401/// ```
402///
403/// ## Async tests
404///
405/// `test_casing` supports all kinds of async test wrappers, such as `async_std::test`,
406/// `tokio::test`, `actix::test` etc. The corresponding attribute just needs to be specified
407/// *below* the `test_casing` attribute.
408///
409/// ```
410/// # use test_casing::test_casing;
411/// # use std::error::Error;
412/// #[test_casing(3, ["0", "42", "-3"])]
413/// #[async_std::test]
414/// async fn parsing_numbers(s: &str) -> Result<(), Box<dyn Error>> {
415///     assert!(s.parse::<i32>()?.abs() < 100);
416///     Ok(())
417/// }
418/// ```

460    /// Strips the repeating module name at the end of `path`.
461    #[cfg(feature = "tracing")]
462    pub const fn parent_module(path: &'static str, test_name: &'static str) -> &'static str {
463        const fn str_eq(x: &str, y: &str) -> bool {
464            if x.len() != y.len() {
465                return false;
466            }
467
468            let x = x.as_bytes();
469            let y = y.as_bytes();
470            let mut i = 0;
471            while i < x.len() {
472                if x[i] != y[i] {
473                    return false;
474                }
475                i += 1;
476            }
477
478            true
479        }
480
481        let Some((head, test_module)) = split_last(path) else {
482            return path;
483        };
484
485        if str_eq(test_module, test_name) {
486            head
487        } else {
488            path
489        }
490    }
491
492    #[cfg(feature = "tracing")]
493    const fn split_last(path: &str) -> Option<(&str, &str)> {
494        let path_bytes = path.as_bytes();
495        let mut i = path_bytes.len().checked_sub(1);
496        let mut found = None;
497        while let Some(pos) = i {
498            if path_bytes[pos] == b':' && path_bytes[pos + 1] == b':' {
499                found = Some(pos);
500                break;
501            }
502            i = pos.checked_sub(1);
503        }
504
505        if let Some(pos) = found {
506            let (head_bytes, tail_bytes) = path_bytes.split_at(pos);
507            let (_, tail_bytes) = tail_bytes.split_at(2); // remove `::` at the start
508            Some(unsafe {
509                // SAFETY: `path_bytes` is a valid UTF-8 string, and we split it at a char boundary.
510                (
511                    core::str::from_utf8_unchecked(head_bytes),
512                    core::str::from_utf8_unchecked(tail_bytes),
513                )
514            })
515        } else {
516            None
517        }
518    }
